« Tool » — plateforme d'évaluation générique pilotée par chatbot

Proposition de valeur

  • Polyvalence — un même socle évalue processus, systèmes et ressenti client via des templates dédiés.
  • Vitesse — du chargement de données au score en quelques clics ; le chatbot remplace la construction manuelle de rapports.
  • Transparence — chaque critère décrit ce qu'il mesure, avec un périmètre « couvert / non couvert » explicite.
  • Souveraineté — modèle IA local : aucune donnée client ne quitte l'infrastructure.
  • Intégrabilité — API REST documentée pour brancher l'outil dans une autre application.

Principes directeurs

  • API-first — l'API est le seul contrat ; l'UI n'est qu'un client parmi d'autres.
  • Découplage IA — le modèle est derrière /api/v1/chat, donc interchangeable (Mistral → Llama → autre).
  • Embeddable — frontend et widget chatbot intégrables dans une application hôte.
  • Stateless API — la logique d'inférence ne stocke pas d'état ; scalable horizontalement.

Le modèle d'évaluation

Chaque template = un ensemble de critères pondérés, notés sur une échelle, agrégés en un score puis traduits en grade. La transparence du périmètre est un parti pris produit.

Processus template 1

Évalue la maturité et l'efficacité d'un processus métier.

Couvert
  • Étapes & enchaînement
  • Goulots & délais
  • Conformité aux règles
Non couvert
  • Ressenti utilisateur → Sentiment
  • Infra technique → Système

Système template 2

Évalue la qualité technique d'un système ou applicatif.

Couvert
  • Fiabilité & disponibilité
  • Performance technique
  • Sécurité & maintenabilité
Non couvert
  • Process organisationnel → Processus
  • Satisfaction → Sentiment

Sentiment client template 3

Évalue le ressenti et la satisfaction d'un client.

Couvert
  • Tonalité & satisfaction
  • Points de friction perçus
  • Verbatim / qualitatif
Non couvert
  • Causes process → Processus
  • Causes techniques → Système

Mécanique de notation

  • Critères — chaque template porte une liste de critères, chacun avec une description fine de ce qui est mesuré.
  • Échelle — notation par curseur (0 → max) par critère.
  • Pondération — chaque critère a un poids ; le score est une moyenne pondérée normalisée.
  • Score → grade — le score numérique est mappé sur un grade lisible (ex. A / B / C / D).
  • Ticket — résultat figé : template, score, grade, détail par critère, périmètre.

Grille de grades (proposée)

A — Excellent85 – 100
B — Bon70 – 84
C — À surveiller55 – 69
D — Insuffisant40 – 54
E — Critique< 40

Seuils paramétrables par template selon le contexte métier.

Le chatbot dans l'évaluation

Le chatbot reçoit le contexte des évaluations (templates, critères, tickets) et permet d'interroger les résultats en langage naturel : « quel processus a le grade le plus bas ? », « résume les points de friction des 3 derniers tickets client », « quels critères font baisser le score système ? ». Il accélère l'interprétation sans imposer la construction manuelle de rapports.

Client & expérience

Deux usages : l'évaluateur qui pilote l'outil, et le client final à qui l'on transmet des données ou un ticket. Le maître-mot reste « rapide et simple ».

Personas & usages

  • L'évaluateur interne — choisit un template, note les critères, génère le ticket, interroge le chatbot.
  • Le client final — reçoit un ticket PDF lisible, ou réalise lui-même une évaluation de ses données.
  • L'application hôte — consomme l'API pour intégrer l'évaluation dans son propre produit.

Parcours type (self-service)

  • 1. Choisir le template adapté (processus / système / sentiment).
  • 2. Charger les données ou remplir le formulaire — chaque critère est expliqué.
  • 3. Obtenir score + grade instantanément.
  • 4. Affiner via le chatbot, puis exporter (PDF / JSON) ou pousser par API.

Sorties client

  • PDF — score, grade, détail des critères + périmètre, lisible par un non-expert.
  • JSON — pour automatiser / réinjecter ailleurs.
  • API — récupération programmatique des tickets.

Principes UX

  • Zéro courbe d'apprentissage — templates guidés.
  • Transparence — on sait toujours ce qui est évalué / pas évalué.
  • Langue — interface et IA en français.

Confiance

  • Données privées — traitées en local, rassurant pour le client.
  • Traçabilité — chaque ticket est daté et reproductible.
  • Périmètre explicite — pas de fausse promesse sur ce qui est mesuré.

Performance

« Rapide et efficace » et « disponible rapidement » sont des exigences produit. On les décline en objectifs mesurables sur l'UI, l'API et l'inférence IA.

<1 s
calcul du score / grade (côté client)
<200 ms
latence API REST (hors chatbot)
<3 s
première réponse du chatbot (streaming)

Leviers de performance

  • Scoring côté client — le calcul reste instantané, sans aller-retour serveur.
  • Réponses en streaming — le chatbot affiche les tokens au fil de l'eau pour une latence perçue faible.
  • Quantization du modèle — Mistral 7B quantifié (Q4) pour tourner sur un GPU modeste, voire CPU.
  • Cache — templates et questions fréquentes mis en cache.
  • PDF serveur — WeasyPrint en tâche dédiée pour ne pas bloquer l'API.

Scalabilité & disponibilité

  • API stateless — réplicable horizontalement derrière un load balancer.
  • Inférence isolée — Ollama sur un nœud dédié, dimensionné indépendamment.
  • File d'attente — si la charge IA grimpe, mise en file plutôt que rejet.
  • Démarrage rapide — image Docker prête, « disponible rapidement » par design.

Arbitrage clé : modèle local vs cloud

Le choix open source local privilégie la souveraineté des données et le coût marginal nul par requête, au prix d'un investissement matériel (GPU) et d'une latence un peu supérieure au cloud. Le plan B Llama 3 8B offre plus de qualité contre plus de ressources. L'API /chat isole ce choix : on peut basculer sans toucher au reste.

Sécurité & conformité

L'outil manipule des données métier et clients sensibles, et s'expose par API. La sécurité se joue sur la donnée, l'accès à l'API et l'isolation de l'IA.

Protection des données

  • Traitement local — l'IA open source tourne on-premise : aucune donnée client n'est envoyée à un tiers.
  • Chiffrement — TLS en transit, chiffrement au repos pour la base de tickets.
  • Minimisation — ne stocker que le nécessaire ; purge/anonymisation configurable.
  • RGPD — base légale, droit à l'effacement, journal des traitements.

Sécurité de l'API

  • Authentification — clés API par client / OAuth2 pour l'app hôte.
  • Autorisation — cloisonnement des tickets par tenant.
  • Rate limiting — quotas pour protéger l'inférence IA.
  • Validation — Pydantic valide chaque entrée ; pas d'injection via les formulaires.

Registre des risques

ÉLEVÉ
Fuite de données client
Données sensibles transmises pour évaluation. Mitigation : inférence locale, chiffrement, cloisonnement par tenant, accès journalisé.
ÉLEVÉ
API exposée sans contrôle
Endpoint d'intégration accessible. Mitigation : clés/OAuth2 obligatoires, rate limiting, HTTPS strict, CORS restreint.
MOYEN
Injection de prompt dans le chatbot
Entrées libres vers l'IA. Mitigation : séparation contexte/instruction, garde-fous sur les actions, pas d'exécution d'effets de bord depuis le chat.
MOYEN
Hallucination dans l'interprétation
Le chatbot peut surinterpréter un score. Mitigation : réponses ancrées sur les tickets réels, citation des critères, mention du périmètre.
FAIBLE
Déni de service sur l'inférence
Saturation du modèle. Mitigation : file d'attente, quotas, nœud IA isolé et autoscalable.
